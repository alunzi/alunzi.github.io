Jenkins+MSBuild+Svn搭建持续集成环境Part 3: Jenkins的搭建

Posted on 2020-01-15 | In 知识沉淀 | |
Reading time ≈ 5

前面两节主要讲了MSBuild的使用与创建，那么终于到了最后一个环节就是Jenkins的搭建。Jenkins是一款由Java编写的开源的持续集成工具。Jenkins作为一个平台，可以安装各种各样的插件，集成了持续集成和持续交付工具链中几乎所有的工具。Jenkins在整个生命周期中管理和控制软件交付流程，包括构建，文档，测试，打包，阶段，部署，静态代码分析等等。

Step 1 | 安装Jenkins

点 下载Jenkins安装，安装之后，会默认打开http://localhost:8080/，第一次进去，选择推荐的插件安装，然后进入到主界面。

Step 2 | 安装MSBuild和Email插件

在管理界面主页，点击Mange Jenkins => Manage Plugins

然后切换到Available，通过搜索找到MSBuild Plugin和Email Extension Plugin两个插件，勾选，点击底部的 Install without restart。

回到管理界面主页，点击Mange Jenkins => Global Tool Configuration

找到MSBuild

设置MSBuild的路径，Name随意，然后保存。

Step 3 | 配置邮件通知

同样是主页，点击Manage Jenkins => Configure System

向下滚动找到Extended E-mail Notification，然后点击Advanced…，展开高级设置，设置发件人信息

然后设置邮件内容模板

附上邮件模板内容，可以随意修改：

最底下Jenkins默认的E-mail Notification跟上面同样的设置

设置邮件默认触发机制

最后，点保存。

Step 3 | 创建自动构建任务

回到管理界面主页，点击New Item

Item name随意，根据你的项目来命名，到时候默认工作空间就叫这个名字，然后选择Freestyle project

创建好Item之后，切换到Source Code Management，配置Svn

然后切换到Build，添加构建步骤，我这边添加了两个步骤，第一个步骤，Build a Visual Studio project or solution Using MSBuild选择我们在上一节 Jenkins+MSBuild+Svn搭建持续集成环境Part 2: 创建NopCommerce的MSBuild构建脚本 中创建好的构建脚本，然后第二个步骤，添加批处理命令，首先将构建好的所有文件覆盖到iis程序目录，接着杀掉w3wp进程或者iisreset，当然也可以选择直接在MSBuild构建脚本里面通过创建Task去覆盖发布，但我觉得构建跟覆盖发布是两回事，不要掺和在一起比较好，附上批处理脚本：

切换到Post-build Actions，设置收件人列表和回复人列表，可以用逗号隔开，然后设置邮件内容类型，选择HTML，默认邮件主题与默认邮件内容，都用默认的就行。

往下滚动，跟着设置，Attach Build Log附加构建日志，随意选择，Triggers选择Always

切换到Build Triggers，设置构建触发机制，我这边选择Poll SCM，这意思是通过轮询，定时检查源码变更（变更依据是软件的版本号），如果轮询发现有变更，就会Checkout下来自动构建。Schedule是设置时间间隔，格式跟Cron语法类似，H/5 * * * *表示5分钟检查一次，最后点击保存。

至此，Jenkins的配置已经全部完成。当源码提交Svn之后，Jenkins发现程序版本号更改那么就会自动构建，构建完之后会自动发送邮件通知团队成员。自从有了自动构建，整个世界都美好了。
